## 二、技术准备（最关键）

### 2.1 网络

| 事项 | 要求 | 验证方式 |
|------|------|---------|
| 教室WiFi | 稳定，能访问 Claude/DeepSeek API | `curl -I https://api.anthropic.com` |
| 备用热点 | 讲师手机热点测试通过 | 实际连接并跑一次 API 调用 |
| 防火墙 | 未封锁 GitHub、npm、pip | `git clone https://github.com` |

### 2.2 软件环境（每台机器）

```bash
# 学员机器和讲师机器都需要以下：

# Claude Code（主力工具）
npm install -g @anthropic-ai/claude-code

# OpenCode（可选，用于成本对比演示）
npm install -g @opencode-ai/cli

# Git + GitHub CLI
brew install gh    # macOS
# 或 apt install gh  # Linux

# Node.js >= 18（Claude Code 前置）
node -v   # 应 >= v18.0
```

### 2.3 API Key 配置

```bash
# 至少需要两种模型（用于成本对比演示）

# Claude（主力 + 复杂推理）
export ANTHROPIC_API_KEY=sk-ant-xxx

# DeepSeek（便宜模型，日常任务用）
export DEEPSEEK_API_KEY=sk-xxx

# 关闭扩展思考（立即省 20% Token）
export CLAUDE_CODE_ENABLE_THINKING=false

# 开启自动压缩
export CLAUDE_CODE_AUTO_COMPACT=true
```

### 2.5 Token 数据准备

```bash
# 检查讲师机器的真实 Token 消耗数据
cat ~/.claude/usage.json

# 准备两段数据用于模块二"Token震撼教育"：
#   1. "没优化"的账单：一个大模块一次会话的消耗
#   2. "优化后"的账单：同一模块用分模块+compact后的消耗

d = json.load(sys.stdin)
total = sum(v for v in d.values() if isinstance(v,(int,float)))
print(f'总Token消耗: {total:,}')
"

## 四、互动环节准备

### 4.1 模块一：痛点便签墙

**准备**：
- 3 色便签纸 + 白板
- 白板上画 3 列：Token 太贵 / AI 代码不可信 / 不知道何时用 AI
- 每人到教室时领 1 张便签，写上最大痛点贴在对应列

**讲师话术**：
> "你上次用 AI 写代码，最大的痛点是什么？写一个关键词贴在白板上。课程结束时我们回来看解决了哪些。"

---

### 4.2 模块三：Prompt 对决

**准备**：
- 选一个简单的 Android Activity 文件作为 Prompt 目标（建议：`LoginActivity.kt`，约 200 行）
- 准备 3 个"烂 Prompt"作为反面教材
- 准备讲师版本的"最优 Prompt"作为对照

**烂 Prompt 示例**（用于展示 Level 0→Level 3 进化）：
```
Level 0: "把这个文件转成鸿蒙的" + 粘贴全部代码
Level 1: "把 LoginActivity 迁移到 ArkTS"
Level 2: "翻译 @LoginActivity.kt:45-200 的 UI 到 ArkTS，只输出代码"
Level 3: "参考 LoginPage 模式迁移 HomePage，API映射见 CLAUDE.md"
```

**评分标准**（学员互评 + 讲师点评）：
| 维度 | 权重 | 说明 |
|------|:--:|------|
| 代码可直接编译 | 30% | 拷贝到项目就能 build |
| 功能完整 | 30% | 与原文件功能对等 |
| Token 消耗 | 20% | Prompt 越省 Token 越好 |
| 风格一致 | 10% | 与项目现有代码风格匹配 |
| 错误处理 | 10% | 边界条件和异常处理 |

---

### 4.3 模块四：代码审查游戏

**准备**：3 段 AI 生成的代码（每段 30-50 行，同一功能不同质量）

**代码 A** — "看起来完美但有设计幻觉"：
```
需求: 计算器百分比功能
AI 输出: 实现了复杂的比例计算引擎，包括盈亏分析、同比增长...
问题: 实现了需求里不存在的功能，过度设计
```

**代码 B** — "功能 OK 但边界缺失"：
```
需求: 计算器除零处理
AI 输出: 实现了除法，但没处理除零情况，没检查负数
问题: 正常输入 OK，边缘输入崩溃
```

**代码 C** — "简洁但有安全风险"：
```
需求: 用户输入验证
AI 输出: 只检查了长度，没校验注入字符
问题: 可能被注入恶意字符
```

> 📌 三段代码需提前在实际项目中跑过并能复现问题。

**审查清单**（打印给学员）：
```
🔴 必查（每次必做）:
□ 功能完整: 是否实现了Spec描述的所有功能？
□ 边界处理: 空输入/极值/异常输入都处理了？
□ 安全校验: 输入校验/权限检查/敏感数据处理？
□ 编译通过: 实际编译过吗？
□ 错误提示: 中文错误提示是否准确？

🟡 选查（复杂逻辑时做）:
□ 性能: 有没有不必要的循环/重复计算？
□ 可维护: 命名是否合理？逻辑是否清晰？
□ 架构: 是否遵循项目的分层约定？
```

```

## 六、应急方案

| 问题 | 方案 |
|------|------|
| 教室网络故障 | 播放备用录屏视频；讲师开手机热点 |
| Claude API 限流 | 切换到 DeepSeek 演示；解释不同模型的选择策略 |
| 学员机器未装 Claude Code | 提前发安装指南；教室 WiFi 可下载；准备 U 盘离线包 |
| 演示项目编译失败 | 提前录制成功的录屏；切换到另一个已跑通的演示 |
| 学员人数超出预期 | 备好电子版所有材料，分组从 4 组扩到 6 组 |
| 投影仪故障 | 提前准备 HDMI 转接头；讲师屏幕共享到 TV/投屏 |
