## 三、各模块详细设计

### L1: AI Native 开发范式 (45min + 15min 演示)

**目标**: 建立AI Native开发的正确心智模型。

#### 时间分配 (60min)

| 时间 | 环节 | 内容 |
|:---:|------|------|
| 5min | 问题抛出 | "你上次用AI写代码踩了什么坑？" — 收集学员痛点贴便签 |
| 15min | 三层模型讲解 | 传统开发 → AI增强 → AI Native。"规划者/引导者/终审法官"三角色模型 |
| 10min | SDD融合 | 华为SDD规范驱动开发流程如何与AI结合。Spec-First → Agent执行 → 4项验收 |
| 10min | 实战数据 | 展示安卓转鸿蒙真实数据：12模块迁移，Token从$1,440降到$3.50 |
| 🆕 15min | **Live 演示** | 现场对比：同一个简单需求，传统方式 vs AI Native 方式的效率差异 |
| 5min | 小结 | "你不是在写代码，你是在给AI写需求规格书" |

#### 关键教学点

1. **三种开发范式的本质差异**：
   - 传统：人→编辑器→编译器→运行
   - AI增强：人↔AI→编辑器→编译器→运行
   - AI Native：人写Spec→AI执行→人审查

2. **AI Native 三角色模型**：
   ```
   规划者(Planner)  → 写Spec、拆任务、定义验收标准
   引导者(Guide)    → 纠偏、给上下文、优化Prompt
   终审法官(Judge)  → 审查代码、验证、决定是否合入
   ```

3. **SDD + AI 融合点**：
   - 需求阶段：AI辅助生成Spec文档
   - 设计阶段：AI生成架构方案备选
   - 实现阶段：AI执行代码迁移/生成
   - 验证阶段：AI辅助4项验收（编译/验证/修复/问题定位）

4. **真实数据冲击**：
   - 12个模块安卓转鸿蒙迁移
   - 纯手工预估：3人×3个月
   - 纯Opus无优化：$1,440 Token费用
   - 混合模型+优化：$3.50 Token费用 + 2周

#### 互动: 痛点便签墙
- 每人写1张便签贴在白板上，课程结束时回顾

### L2: Agent/Claude架构深度解读 (60min + 15min 演示)

**目标**: 理解Agent如何工作，掌握Token消耗和上下文管理。

#### 时间分配 (75min)

| 时间 | 环节 | 内容 |
|:---:|------|------|
| 10min | 架构全景 | Agent = 上下文管理 + Skills技能 + 记忆存储 + 驾驭工程 |
| 15min | Token消耗拆解 | 系统提示词20-30K/次、文件读取30-50%、对话历史15-30% |
| 15min | 上下文管理 | /compact机制(200K→55.7K)、分模块会话、Prompt Caching(90%折扣) |
| 🆕 15min | **Live 演示** | 打开Claude Code，展示/tokens、/context、/compact + 消耗前后对比 |
| 10min | 动手 | 学员在自己的终端检查token消耗 |
| 10min | 总结 | Claude Code vs OpenCode 选型 + 开源协调 |

#### 关键教学点

1. **Agent 的四层架构**：
   ```
   上下文管理  → 拆解任务为可执行步骤
   Skills技能  → 可复用的能力封装
   记忆存储    → 存储项目约定(CLAUDE.md) + 会话历史
   驾驭工程    → 工具调用(文件读写/编译/搜索)
   ```

2. **Token消耗全景**：
   | 消耗源 | 占比 | 典型量 | 优化手段 |
   |--------|:--:|------|---------|
   | 系统提示词 | 20-30% | 20-30K/次 | 不可控（起步价） |
   | 文件读取 | 30-50% | 10-500K/次 | .claudeignore + 方法级引用 |
   | 对话历史 | 15-30% | 累积增长 | /compact + 分模块 |
   | Skills | 5-15% | 5-15K/次 | 精简Skills内容 |

3. **Skills 机制的三个核心原则**：
   - **自包含**：一个 Skill 独立工作，复制到项目目录即用
   - **分层可组合**：Pipeline 编排多个 Skill，Agent 调度执行
   - **知识可验证**：CI 自动校验 Skill 格式，防止知识腐化

4. **Claude Code vs OpenCode 选型**：
   | 场景 | 推荐工具 | 原因 |
   |------|---------|------|
   | 日常编码 | Claude Code | 生态成熟 |
   | 便宜任务 | OpenCode+DeepSeek | 成本1/10 |
   | 架构设计 | Claude Code+Opus | 推理最强 |
   | 代码审查 | OpenCode+Haiku | 性价比最高 |

#### 互动: Token账单震撼教育
- 现场让学员查 `~/.claude/usage.json` 看自己的真实消耗
- 讲师展示"没优化"的账单 vs "优化后"的账单

### L3: 全流程AI Native推演 (90min + 30min 演示)

**目标**: 用完整案例走一遍需求→设计→实现→验证全流程。

#### 时间分配 (120min)

| 时间 | 环节 | 内容 |
|:---:|------|------|
| 5min | 场景设定 | "我们要给OpenCalc加一个单位换算功能" — 一句话需求 |
| 🆕 30min | **Live Coding** | 完整演示：Spec设计 → AI生成代码 → 编译验证。全程实时不剪辑 |
| 15min | 需求分析 | 拆解刚才的演示：AI如何辅助拆解需求、生成Spec |
| 20min | 设计 | AI生成架构方案、API设计、数据流 |
| 30min | 实现 + 审查 | AI执行代码生成 + 展示3段同功能不同质量的AI代码，学员用审查清单打分 |
| 10min | 验证 | AI辅助4项验收（编译/验证/修复/问题定位） |
| 10min | 复盘 | 整个流程的Token消耗、时间花费、代码质量 |

#### 关键教学点

1. **四幕结构**：
   ```
   第一幕 需求 → "给计算器加单位换算" → AI拆解为子任务 + 生成Spec
   第二幕 设计 → AI生成架构方案/API设计/数据流
   第三幕 实现 → AI写代码 → 编译验证 → 修复迭代
   第四幕 验证 → 4项验收（编译/功能验证/边界修复/问题定位）
   ```

2. **SDD 在每个环节的体现**：
   - **需求**: Spec文档（AI辅助生成，人工确认）
   - **设计**: 架构决策记录ADR（AI生成备选，人工选择）
   - **实现**: 原子任务分解（AI执行，人工审查）
   - **验证**: 4项验收标准

3. **Spec-as-Code 理念**：
   - 规格即代码，代码是 Spec 的编译产物
   - 对比演示：好的 Spec → AI 一次生成编译通过 vs 差的 Spec → 反复修正

4. **代码审查游戏**：
   - 讲师展示3段AI生成的代码（同功能不同质量）
   - 代码A：看起来完美，但有设计幻觉
   - 代码B：功能OK，但边界条件缺失
   - 代码C：简洁但引入了安全风险

#### 互动: 代码审查游戏
- 每人用审查清单打分：功能完整/边界处理/安全/代码风格
- 组内讨论 → 讲师揭晓每段代码的隐藏问题

### L4: 安卓转鸿蒙实战案例 (50min)

**目标**: 基于真实客户项目，深入解读安卓转鸿蒙的实战过程和经验教训。

#### 时间分配 (50min)

| 时间 | 环节 | 内容 |
|:---:|------|------|
| 10min | 案例概览 | 项目背景、模块规模、迁移策略 |
| 15min | Skill 机制 | 自包含 → 分层组合 → CI验证 → 多平台适配 |
| 15min | 实战过程 | 迁移铁律、Token 优化实战、踩坑记录 |
| 10min | 经验总结 | 可复用的模式、反面教材、决策框架 |

#### 关键教学点

1. **Skill 的四个核心原则**：
   ```
   自包含      → 独立工作，复制到项目目录即用
   分层可组合  → Pipeline 编排、Agent 调度 Skill
   知识可验证  → CI 自动校验格式、防止知识腐化
   多平台适配  → 适配层处理平台差异（Android API → ArkUI API）
   ```

2. **迁移铁律**（来自真实项目）：
   - ① Spec-First → 原子任务 → 编译验证
   - ② done/total checklist，禁止自评百分比
   - ③ 4项验证全过（编译/验证/修复/定位）
   - ④ 深度优先优于广度优先
   - ⑤ 禁止子代理写 ArkTS 代码
   - ⑥ API 行为对等 > 代码结构优雅

3. **Token 优化实战成果**：
   | 阶段 | Token 费用 | 优化手段 |
   |------|:--:|------|
   | 初版（无优化） | $1,440 | 全模块一次会话 |
   | 优化第一轮 | $120 | 分模块会话 + /compact |
   | 优化第二轮 | $12 | 方法级引用 + Prompt Caching |
   | 最终方案 | $3.50 | 混合模型 (70% DeepSeek + 25% Sonnet + 5% Opus) |

4. **反面教材 — 典型错误**：
   - ❌ 粘贴全文让 AI 迁移 → Token 爆了，结果不可用
   - ❌ 不加 CLAUDE.md → AI 不知道项目规范，风格混乱
   - ❌ 不拆任务 → 一个 session 做到一半上下文溢出
   - ✅ 正确做法：Spec + 原子任务 + 分批迁移

#### 互动: 迁移决策讨论
- 给一个 Android Activity 文件，学员组讨论：怎么拆？用什么模型？
- 各组写下迁移计划，讲师点评

### L5: 工程最佳实践 (50min)

**目标**: 建立 AI 代码的可信度评估框架，掌握效率/质量/安全/成本的平衡。

#### 时间分配 (50min)

| 时间 | 环节 | 内容 |
|:---:|------|------|
| 15min | 效率 | 用更少资源、更短时间达成目标。70/30 法则（70%任务用便宜模型） |
| 15min | 质量 | 确保产出可靠、可维护、符合预期。AI代码审查清单 |
| 15min | 安全可信 | 数据安全、符合规范。AI 代码三重门禁 |
| 5min | 成本 | 资源投入与长期效益的平衡小结 |

#### 关键教学点

1. **效率 — 70/30 模型分配法则**：
   | 任务类型 | 模型 | 占比 |
   |---------|------|:--:|
   | UI迁移/格式化/重命名 | DeepSeek V4 | 70% |
   | 逻辑迁移/API对接 | Claude Sonnet | 25% |
   | 架构设计/复杂审查 | Claude Opus | 5% |

2. **质量 — AI 代码审查清单**：
   ```
   🔴 必查 (每次必做):
   □ 功能完整 □ 边界处理 □ 安全校验 □ 编译通过 □ 错误提示
   🟡 选查 (复杂逻辑时做):
   □ 性能 □ 可维护 □ 架构
   ```

3. **安全可信 — AI 代码三重门禁**：
   ```
   第一道：自动测试    → 编译 + 单元测试 + lint
   第二道：人工审查    → AI代码审查清单逐项检查
   第三道：安全检查    → 输入校验/权限/敏感数据处理
   ```

4. **决策矩阵：何时信AI**：
   ```
   高风险 × 复杂逻辑 → 🔴 人工写
   高风险 × 简单逻辑 → 🟡 AI写 + 严格审查
   低风险 × 复杂逻辑 → 🟡 AI写 + 抽查
   低风险 × 简单逻辑 → 🟢 AI写 + 编译验证即可
   ```

#### 关键教学点

1. **Token优化三步法**：
   ```
   今天(10分钟): 关闭扩展思考 + 开启自动压缩
   ```
