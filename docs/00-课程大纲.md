## 三、各模块详细设计

### 模块一：AI Native 开发范式变革 (45min)

**目标**: 建立AI Native开发的正确心智模型，破除"AI就是自动补全"的误解。

#### 时间分配 (45min)

| 时间 | 环节 | 内容 |
|:---:|------|------|
| 5min | 问题抛出 | "你上次用AI写代码踩了什么坑？" — 收集学员痛点贴便签 |
| 15min | 三层模型讲解 | 传统开发 → AI增强 → AI Native。引入"规划者/引导者/终审法官"三角色模型 |
| 10min | SDD融合 | 华为SDD规范驱动开发流程如何与AI结合。Spec-First → Agent执行 → 5项验收 |
| 10min | 实战数据 | 展示安卓转鸿蒙真实数据：12模块迁移，Token从$1,440降到$3.50 |
| 5min | 小结 | "你不是在写代码，你是在给AI写需求规格书" |

#### 关键教学点

1. **三种开发范式的本质差异**：
   - 传统：人→编辑器→编译器→运行
   - AI增强：人↔AI→编辑器→编译器→运行（AI是副驾驶）
   - AI Native：人写Spec→AI执行→人审查（AI是执行者，人是决策者）

2. **AI Native的三角色模型**：
   ```
   规划者(Planner)  → 写Spec、拆任务、定义验收标准
   引导者(Guide)    → 纠偏、给上下文、优化Prompt
   终审法官(Judge)  → 审查代码、验证、决定是否合入
   ```

3. **SDD + AI 融合点**：
   - 需求阶段：AI辅助生成Spec文档
   - 设计阶段：AI生成架构方案备选
   - 实现阶段：AI执行代码迁移/生成
   - 验证阶段：AI辅助5项验收

4. **真实数据冲击**（来自我们的项目）：
   - 12个模块安卓转鸿蒙迁移
   - 纯手工预估：3人×3个月
   - 纯Opus无优化：$1,440 Token费用
   - 混合模型+优化：$3.50 Token费用 + 2周

#### 互动: 痛点便签墙
- 每人写1张便签："AI Coding最大的痛点"
- 贴在白板上，课程结束时回顾哪些被解决

### 模块二：Agent架构深度解读 (60min)

**目标**: 理解Agent如何工作，知道Token花在哪，学会控制成本和上下文。

#### 时间分配 (60min)

| 时间 | 环节 | 内容 |
|:---:|------|------|
| 10min | 架构全景 | Agent = Planner + Memory + Skills + Harness |
| 15min | Token消耗拆解 | 系统提示词20-30K/次、文件读取30-50%、对话历史15-30%。展示真实usage.json |
| 15min | 上下文管理 | /compact机制(200K→55.7K)、分模块会话、Prompt Caching(90%折扣) |
| 10min | 实战演示 | 现场打开Claude Code，展示/tokens、/context、/compact |
| 10min | 动手 | 学员在自己的终端检查token消耗 |

#### 关键教学点

1. **Agent的四层架构**：
   ```
   Planner    → 拆解任务为可执行步骤
   Memory     → 存储项目约定(CLAUDE.md) + 会话历史
   Skills     → 可复用的能力封装
   Harness    → 工具调用(文件读写/编译/搜索)
   ```

2. **Token消耗全景**（基于真实项目数据）：
   | 消耗源 | 占比 | 典型量 | 优化手段 |
   |--------|:--:|------|---------|
   | 系统提示词 | 20-30% | 20-30K/次 | 不可控（起步价） |
   | 文件读取 | 30-50% | 10-500K/次 | .claudeignore + 方法级引用 |
   | 对话历史 | 15-30% | 累积增长 | /compact + 分模块 |
   | MCP Server | 5-15% | 5-15K/次 | 关掉闲置的 |

3. **Claude Code vs OpenCode 选型**：
   | 场景 | 推荐工具 | 原因 |
   |------|---------|------|
   | 日常编码 | Claude Code | 生态成熟 |
   | 便宜任务 | OpenCode+DeepSeek | 成本1/10 |
   | 架构设计 | Claude Code+Opus | 推理最强 |
   | 代码审查 | OpenCode+Haiku | 性价比最高 |

4. **开源协调**：Prompt+LLM模型+Coder模型+传统开发工具

#### 互动: Token账单震撼教育
- 现场让学员运行 `cat ~/.claude/usage.json` 看自己的真实消耗
- 讲师展示一个"没优化"的账单 vs "优化后"的账单

### 模块三：Prompt工程 — 精准操控术 (80min)

**目标**: 从Attention机制理解Prompt为何有效，掌握CRISPE框架，能写出一句话就让AI正确迁移代码。

#### 时间分配 (80min)

| 时间 | 环节 | 内容 |
|:---:|------|------|
| 10min | 机制基础 | Attention是如何"注意"Prompt的？Token→Embedding→Attention→预测 |
| 15min | CRISPE框架 | Capacity/Role/Insight/Statement/Personality/Experiment 六要素 |
| 20min | 安卓转鸿蒙案例 | 用真实案例演示Prompt从"烂"到"好"的进化 |
| 15min | 动手练习 | 学员写Prompt让AI迁移一个SimpleActivity |
| 10min | 对比评审 | 对比各组结果，分析Prompt差异导致的质量差异 |
| 10min | 高级技巧 | Few-Shot、Chain-of-Thought、编辑原提示词代替追问 |

#### 关键教学点

1. **从Attention理解Prompt**：
   - Attention本质是"相关性加权"——Prompt中的每个词都在竞争模型的注意力
   - 冗余词稀释注意力 → 结果变差
   - 精确词聚焦注意力 → 结果变好
   - **一句话原则**：Prompt越短越精确，效果越好

2. **CRISPE框架**（迁移项目适配版）：
   ```
   Capacity    → "你是Android→鸿蒙迁移专家"
   Role        → "你的任务是准确翻译代码逻辑"
   Insight     → "参考 @LoginActivity.kt:45-200 的UI布局"
   Statement   → "迁移到ArkTS，保持功能对等"
   Personality → "只输出代码，不要解释"
   Experiment  → "如果API不存在，用最接近的替代并标注TODO"
   ```

3. **Prompt进化演示**（真实案例）：
   ```
   Level 0（新手）: "帮我迁移这个文件" + 粘贴全文
   → Token: 15K  | 结果: 不可用

   Level 1（入门）: "迁移LoginActivity到鸿蒙，参考 @LoginActivity.kt"
   → Token: 8K   | 结果: 可用但需大量修改

   Level 2（熟练）: "翻译 @LoginActivity.kt:45-200 的UI布局到ArkTS。只输出代码。"
   → Token: 3K   | 结果: 直接用

   Level 3（专家）: "参照LoginPage的迁移模式，翻译HomePage。API映射见CLAUDE.md"
   → Token: 1K   | 结果: 直接用 + 风格一致
   ```

4. **编辑原提示词代替追问**（新技巧）：
   - 不要追加"这里改一下""那里不对"
   - 直接编辑第一条提示词，把修改需求补充进去，重新生成
   - 省50% Token，结果更准

#### 互动: Prompt对决
- 给出同一个Android文件
- 3组写不同的Prompt让AI迁移
- 投票评出"最强Prompt"（代码质量最好 + Token最少）
- 讲师分析差异原因

### 模块四：全流程AI协同推演 (90min)

**目标**: 用"电影剧本"方式完整走一遍需求→设计→实现→测试全流程，让学员看到AI在每个环节的具体用法。

#### 时间分配 (90min)

| 时间 | 环节 | 内容 |
|:---:|------|------|
| 5min | 场景设定 | "我们要给OpenCalc加一个单位换算功能" — 一句话需求 |
| 15min | 第一幕: 需求分析 | AI辅助拆解需求、生成Spec、评估工作量 |
| 20min | 第二幕: 设计 | AI生成架构方案、API设计、数据流 |
| 30min | 第三幕: 实现 | AI执行代码生成、学员审查、编译验证 |
| 10min | 第四幕: 测试 | AI生成测试用例、学员运行验证 |
| 10min | 复盘 | 整个流程的Token消耗、时间花费、代码质量 |

#### 关键教学点

1. **"电影剧本"五幕结构**：
   ```
   第一幕 需求 → "给计算器加单位换算" → AI拆解为7个子任务
   第二幕 设计 → AI生成: 换算类别/API设计/UI布局
   第三幕 实现 → AI写代码 → 人审查 → 编译 → 修复
   第四幕 测试 → AI生成测试用例 → 人运行 → 通过
   第五幕 复盘 → Token花了多少？代码质量如何？学到了什么？
   ```

2. **SDD在每个环节的体现**：
   - 需求: Spec文档（AI辅助生成）
   - 设计: 架构决策记录ADR（AI生成备选）
   - 实现: 原子任务分解（AI执行）
   - 验证: 5项验收标准（编译/状态/回滚/错误/空态）

3. **代码审查游戏**（核心互动）：
   - 讲师展示3段AI生成的代码（同功能不同质量）
   - 学员用"AI代码审查清单"打分
   - 揭露每段代码的隐藏问题：
     - 代码A：看起来完美，但有设计幻觉（实现了一个不存在的需求）
     - 代码B：功能OK，但边界条件缺失（没处理空输入）
     - 代码C：简洁但引入了安全风险（没做输入校验）

4. **实际操作**：
   - 学员在自己的终端用Claude Code实现"单位换算"
   - 限时15分钟
   - 评出最快完成 + 代码最规范的

#### 互动: 代码审查游戏
- 讲师展示3段代码（A/B/C）
- 每人用审查清单打分：功能完整/边界处理/安全/代码风格
- 组内讨论 → 组间辩论
- 讲师揭晓：哪段是AI写的，哪段是人写的，哪段是混合的

### 模块五：工程判断力 — 质量、安全与成本 (50min)

**目标**: 建立AI代码的可信度评估框架，知道什么时候信AI，什么时候必须人工介入。

#### 时间分配 (50min)

| 时间 | 环节 | 内容 |
|:---:|------|------|
| 10min | AI常见错误类型 | 设计幻觉/边界缺失/安全漏洞/过度工程/风格不一致 |
| 15min | AI代码审查清单 | 7项必查 + 3项选查 |
| 10min | 决策模型 | 什么时候信AI？什么时候人工写？— 二维决策矩阵 |
| 10min | Token成本权衡 | 什么时候用便宜模型？什么时候必须上Opus？ |
| 5min | 小结 | "工程判断力 = 经验 + 清单 + 敢于不信" |

#### 关键教学点

1. **AI的五种"高级错误"**：
   | 错误类型 | 特征 | 案例 |
   |---------|------|------|
   | 设计幻觉 | 实现了不存在的需求 | 给计算器加了"积分系统" |
   | 边界缺失 | 正常输入OK，边缘崩溃 | 没处理除零、负数阶乘 |
   | 安全漏洞 | 代码能用但不安全 | 没校验输入长度 |
   | 过度工程 | 为"灵活性"加了不必要的抽象 | 3行功能写成30行 |
   | 风格不一致 | 混用不同命名/结构 | camelCase + snake_case混用 |

2. **AI代码审查清单**（核心交付物）：
   ```
   🔴 必查 (每次必做):
   □ 功能完整: 是否实现了Spec描述的所有功能？
   □ 边界处理: 空输入/极值/异常输入都处理了？
   □ 安全校验: 输入校验/权限检查/敏感数据处理？
   □ 编译通过: 实际编译过吗？（AI可能生成语法正确的"幻觉代码"）
   □ 错误提示: 中文错误提示是否准确？

   🟡 选查 (复杂逻辑时做):
   □ 性能: 有没有不必要的循环/重复计算？
   □ 可维护: 命名是否合理？逻辑是否清晰？
   □ 架构: 是否遵循项目的分层约定？
   ```

3. **决策矩阵：何时信AI**：
   ```
   高风险 × 复杂逻辑 → 🔴 人工写
   高风险 × 简单逻辑 → 🟡 AI写 + 严格审查
   低风险 × 复杂逻辑 → 🟡 AI写 + 抽查
   低风险 × 简单逻辑 → 🟢 AI写 + 编译验证即可

   判断标准:
   风险 = 涉及资损？涉及安全？影响面大？
   复杂度 = 跨多个模块？需要深度推理？
   ```

4. **Token成本决策**：
   | 任务 | 模型 | 月费 | 比例 |
   |------|------|:---:|:--:|
   | UI迁移/格式化/重命名 | DeepSeek V4 | $15 | 70% |
   | 逻辑迁移/API对接 | Claude Sonnet | $18 | 25% |
   | 架构设计/复杂审查 | Claude Opus | $20 | 5% |

#### 互动: "信还是不信"决策游戏
- 讲师快速展示5个场景（如"登录模块的安全校验"）
- 学员举牌：🟢信AI / 🟡半信 / 🔴人工
- 每组派代表辩论
- 讲师公布答案并解释理由

### 模块六：落地策略 + 总结 (40min)

**目标**: 学员带着可操作的计划离开，第二天就能开始用。

#### 时间分配 (40min)

| 时间 | 环节 | 内容 |
|:---:|------|------|
| 5min | Token优化速查 | 一页纸总结"今天/本周/持续"三步法 |
| 10min | 团队推行策略 | 友好PDU先行、试点项目选择、效果度量 |
| 10min | 个人学习路径 | 30天从新手到熟练的路线图 |
| 10min | 回顾痛点墙 | 回顾开场时的痛点便签，检查哪些已被解决 |
| 5min | Q&A + 结束 | 发放材料 + 问卷 |

#### 关键教学点

1. **Token优化三步法**（可打印速查卡）：
   ```
   今天(10分钟):
   export CLAUDE_CODE_ENABLE_THINKING=false
   export CLAUDE_CODE_AUTO_COMPACT=true
   
   本周:
   创建 .claudeignore | 精简 CLAUDE.md | 配置 DeepSeek
   
   持续: 
   每15轮 /compact | 一模块一会话 | 简单任务用便宜模型
   ```

2. **团队推行"4步走"**：
   - Week 1: 选1-2个友好PDU，配一个简单模块试点
   - Week 2: 收集Token消耗数据、代码质量数据
   - Week 3: 内部复盘分享、优化工作流
   - Week 4: 推广到更多团队、纳入日常SDD流程

3. **个人30天学习路线**：
   ```
   Day 1-3: 熟悉Claude Code基础操作 + 写好CLAUDE.md
   Day 4-7: 用AI完成1个模块迁移，记录Token消耗
   Day 8-14: 掌握Prompt技巧，尝试不同模型对比
   Day 15-21: 建立审查习惯，形成自己的决策矩阵
   Day 22-30: 带新人，输出最佳实践文档
   ```

4. **效果度量方案**：
   | 指标 | 基线 | 培训后7天目标 |
   |------|:--:|:---------:|
   | Token费用/模块 | $20+ | <$5 |
   | 代码审查通过率 | 60% | >85% |
   | AI使用频率 | 偶尔 | 每个任务都用 |
   | 编译一次通过率 | 40% | >70% |

#### 互动: 痛点回顾
- 回到开场时的便签墙
- 逐一检查：哪些痛点已被课程内容解决？
- 未解决的记录下来作为后续优化方向
- 学员写下"明天要做的第一件事"
